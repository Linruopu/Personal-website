## 从 EMS 数据到 Agent 诊断

本文详细讲解 EMU Agent 系统的完整数据管道——从原始工业数据流到结构化诊断报告——解释每个阶段的设计选择。

## 数据全景

电池储能站点的能源管理系统（EMS）生成多类数据：

### 实时遥测
- 电池单体电压（每个电池架 100+ 个测点）
- 温度传感器（环境、单体、PCS）
- 充放电状态（SOC）和健康状态（SOH）
- 功率流（充放电 kW）
- 电网连接状态

### 事件与告警
- 保护事件（过压、欠压、超温）
- 通信故障（Modbus 超时、GOOSE 订阅丢失）
- 设备状态变化（断路器分/合、PCS 启/停）

### 控制命令
- 充放电功率设定值
- 无功功率命令
- 断路器跳闸/闭合命令
- PCS 模式变更

### 历史记录
- 带运维人员注释的历史告警日志
- 维护记录
- 配置变更

所有这些通过工业协议流动——主要是用于轮询数据的 Modbus TCP 和用于事件驱动消息的 IEC61850 GOOSE。

## 阶段一：数据采集与标准化

第一个挑战是协议多样性。一个 C++ 数据采集器处理底层协议工作：

- **Modbus TCP**：以可配置间隔轮询保持寄存器（关键测点 100ms，非关键 1s）
- **IEC61850 GOOSE**：订阅多播 GOOSE 消息获取保护事件
- **IEC61850 MMS**：从 IED（智能电子设备）读取结构化数据

采集器将所有数据标准化为通用 JSON 模式，并通过 FastAPI 服务写入 SQLite。为什么选 SQLite？它是嵌入式的、零维护的，且在 WAL 模式下舒适地处理吞吐量（通常约 10K 数据点/秒）。

## 阶段二：知识库

原始数据不足以进行诊断。系统需要领域知识：

### 故障规范
将故障代码映射到以下内容的结构化表格：
- 严重程度（信息/警告/严重/紧急）
- 可能原因（按可能性排序）
- 推荐措施
- 相关子系统

### 设备文档
- 所有参数的标准运行范围
- 配置模板
- 接线和通信拓扑

### 运维经验
- 带根因注释的历史事件报告
- 解决类似问题的维护流程

这些使用混合方法索引：
- **BM25 关键词搜索**用于精确故障代码匹配
- **稠密嵌入向量**用于症状描述的语义搜索
- **SQL 查询**用于参数范围查找

## 阶段三：AgentGraph 工作流

这就是诊断发生的地方。工作流是一个 DAG，包含以下节点：

### 节点一：意图识别

LLM 调用分类用户意图：
- **告警查询**："电池架 3 出了什么问题？"
- **状态查询**："显示今天的 SOC 趋势"
- **报告请求**："生成本周维护报告"
- **控制意图**："重置 PCS 2 的 Modbus 连接"

每个意图触发不同的子图。

### 节点二：数据检索

基于意图，Agent 查询正确的数据：
- 告警查询 → 指定设备的近期告警 + 告警时刻的实时遥测
- 状态查询 → 当前遥测快照 + 近期趋势
- 报告请求 → 请求时段内的聚合统计

### 节点三：知识检索

由具体场景触发的 RAG 检索：
- 告警的故障代码 → 故障规范中的精确匹配
- 症状描述 → 历史事件的语义搜索
- 设备类型 → 参数范围查找

### 节点四：异常分析

LLM 对标准范围分析数据：
- 是否有参数超出正常范围？
- 告警之间是否存在相关性？
- 模式是否匹配已知的故障模式？

### 节点五：根因归因

最复杂的节点。LLM 综合：
- 告警数据
- 检索到的故障规范
- 相似的历史事件
- 提示词中编码的工程知识

产生带有置信度评分的排序可能根因列表。

### 节点六：报告生成

最终诊断格式化为结构化报告，包含：
- 执行摘要
- 事件时间线
- 分析与根因归因
- 推荐措施
- 参考相关文档

## 阶段四：展示

Vue 3 前端通过 ECharts 可视化呈现报告：
- 告警序列时间线图
- 带异常高亮的参数趋势线
- 诊断推理路径决策树

运维人员可以钻取任何结论以查看底层数据和检索到的文档。

## 为什么这个架构有效

### 关注点分离

数据采集、知识检索和诊断推理是解耦的。你可以在不触及诊断逻辑的情况下改进检索策略，反之亦然。

### 可审计性

每个诊断结论都有追踪：查询了哪些数据、检索了哪些文档、哪些推理步骤导致了结论。在工业场景中，这种可追踪性对运维人员信任至关重要。

### 可扩展性

添加新数据源（例如振动传感器）意味着添加一个新的采集器和新的检索路径——诊断图不变。添加新诊断能力意味着添加一个新的子图。

## 结论

从原始 EMS 数据到 Agent 诊断的旅程涉及每一层的精心工程：协议级数据采集、混合检索索引、基于 DAG 的 Agent 编排以及结构化报告生成。关键洞察是 LLM 只是其中一个组件——用于数据访问、知识检索和工作流编排的周边基础设施对构建一个运维人员可以信任的系统同样重要。
